## 1. Налаштування Продакшну:
### Поясніть, чому небезпечно залишати `DEBUG = True` у файлі `settings.py` на продакшн-сервері. Назвіть щонайменше дві основні загрози.
* При `DEBUG = True`, то Django показує детальні traceback з повною інформацією про сервер, шляхи до файлів і конфігурацію.
* Крім того, статичні файли починає віддавати сам Django, а не сервер. Це робить застосунок повільним і може викликати проблеми з навантаженням.

### Яке призначення змінної ``ALLOWED_HOSTS`` і що відбудеться, якщо не додати до неї доменне ім'я PythonAnywhere?
* `ALLOWED_HOSTS` визначає список доменів, з яких Django дозволяє приймати HTTP запити.
Якщо не додати домен PythonAnywhere - сервер буде повертати помилку Bad Request (400), бо Django вважатиме запит “підозрілим”.

## 2. WSGI vs runserver:
### Поясніть роль WSGI (Web Server Gateway Interface) у продакшн-середовищі Django. Чим він принципово відрізняється від вбудованого сервера python manage.py runserver?
* *WSGI* - це міст між Django та продакшн-сервером. Він оптимізований для продакшну, підтримує багатопроцесність і роботу під великим навантаженням. 
А *runserver* - тільки для розробки, повільний і призначений для реального трафіку.

## 3. Статичні Файли:
### Що робить команда `python manage.py collectstatic` і чому її потрібно виконати на сервері перед запуском застосунку?
* `python manage.py collectstatic` збирає всі статичні файли з додатків і проекту в одну спільну директорію.
Це робиться щоб сервер міг віддавати статику швидко та централізовано.
### Куди саме Django збирає ці файли після виконання команди (яка змінна у settings.py це визначає)?
* Місце збору визначає змінна `STATIC_ROOT`. Django складає всі зібрані файли в цю директорію після виконання ``collectstatic``.
